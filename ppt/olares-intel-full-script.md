## 第 1 页｜个人云与 AI NAS 时代

大家好。

这里先解释一下这两个词。

个人云，是一套运行在用户自己硬件上的计算环境。

它像公有云一样始终在线。

它也能运行应用和 AI。

但硬件和数据属于用户自己。

它适合放在家里或办公室。

它可以同时承担存储、本地 AI 和长期在线服务。

Olares 本身不是 NAS 系统。

Olares 是个人云操作系统。

NAS、AI Box 和个人电脑，都可以是它的硬件载体。

## 第 3 页｜Olares 是什么

这一页的信息比较多。

我分几个部分讲。

先说我们的基本判断。

我们认为，未来很多人都会有自己的 Personal Agent。

它不只是一个聊天工具。

它会帮助用户完成工作。

它会处理日常事务。

它会管理个人信息。

它也会连接用户的各种设备。

用户可能通过手机和它交流。

也可能通过电脑、音箱、眼镜或者手表。

这个 Agent 需要一直运行。

它还需要长期保存用户的上下文。

所以，它需要一个始终在线的运行环境。

我们把这个环境叫作 Personal Cloud。

Personal Cloud 是 Personal Agent 的家。

Olares 是 Personal Cloud 的操作系统。

可以做一个简单的类比。

Windows 运行个人电脑。

Android 运行智能手机。

Olares 运行个人云。

这个类比只解释系统的位置。

Olares 不兼容 Android 应用。

它的技术架构也不等同于 Android。

个人云和个人电脑有几个明显区别。

第一，个人云需要 7×24 小时在线。

第二，它需要从不同地点安全访问。

第三，它需要持续运行后台任务。

第四，它要汇集分散在不同设备和服务里的数据。

第五，它要给多个 AI 任务提供统一的执行环境。

传统个人电脑主要为人在屏幕前操作而设计。

它通常不是一个长期在线的服务环境。

所以，个人云需要不同的系统设计。

为什么我们认为现在是做个人云的时间？

有三个条件正在成熟。

第一个条件，是本地模型。

开源模型正在快速变小。

它们也在快速变强。

一部分去年只能在云端完成的任务，现在已经可以在消费级硬件上完成。

这里不是说本地模型在所有任务上都等于最强云端模型。

更准确的说法是，它已经能满足越来越多具体场景。

第二个条件，是开源应用。

现在已经有不少可以长期使用的开源应用。

比如 Ollama、ComfyUI、Mattermost 和 GitLab。

它们覆盖模型运行、图像生成、团队沟通和代码协作。

这些应用不能完整替代所有公有云服务。

但它们已经能替代其中一部分。

第三个条件，是云原生基础设施。

容器、Kubernetes、服务网格和 GPU 调度，已经在数据中心使用多年。

这些能力现在可以被带到用户自己的硬件上。

Olares 做的事情，是把这三类能力整合起来。

用户不需要自己拼装每一层。

系统负责应用安装、网络、身份、存储、权限和算力分配。

再说用户。

大多数人仍然会继续使用公有云。

因为公有云方便。

它也不需要用户自己管理硬件。

但有一部分高级用户、小企业和家庭，需要另一种选择。

他们在意隐私。

他们在意数据所有权。

他们希望有更低的本地推理延迟。

他们也希望长期成本更可预测。

Olares 主要面向这部分用户。

【可删｜市场空间】

这一页也给出了一个市场空间的估算。

AI 会显著增加个人算力需求。

如果未来全球公有云年收入达到十万亿美元量级。

如果个人云承接其中百分之十。

那么个人云可能形成万亿美元量级的市场。

它不是公司的收入预测。

我们还做了一个时间判断。

过去几年有一个趋势。

前一年最强的云端模型能力，往往会在下一年出现在更小的开源模型上。

幻灯片列了几组模型作为例子。

这些例子不是严格的一一性能等价。

它们表达的是模型能力下沉的方向。

基于这个趋势，我们把 2026 年底看作一个可能的技术拐点。

到 2027 年，更多有商业价值的 AI 工具，可能在消费级硬件上本地运行。

这个时间点是我们的判断。

不是确定的行业结论。

## 第 6 页｜一站式方案与五层架构

这一页左边是用户拿到的产品。

右边是支撑这些产品的系统架构。

我先从右边的最底层讲起。

第一层是 Infrastructure。

可以把它理解成本地的 IaaS。

它负责把硬件变成可调度资源。

这里包括计算、存储、网络、节点和加速器。

代表性组件有 Kubernetes、K3s、containerd、Helm、MinIO 和 HAMi。

这些组件不是卖点本身。

它们的作用，是让上层不需要直接处理每台机器的差异。

第二层是 Platform。

可以把它理解成本地的 PaaS。

它给应用提供共享服务。

比如 PostgreSQL、KVRocks、NATS、JuiceFS、Prometheus 和 Velero。

应用可以复用数据库、消息、文件、监控和备份能力。

它们不需要每个应用都重新搭一套。

第三层是 Framework。

这一层负责应用治理。

应用通过 Manifest 声明需要什么。

比如需要多少资源。

需要哪些存储。

需要什么网络入口。

需要哪些权限。

系统再负责安装、升级、身份认证、单点登录、路由和生命周期。

这一层的角色有些像 Android 的 Application Framework。

但 Olares 不兼容 Android API。

这个类比只用于解释架构分层。

第四层是 Built-in apps。

这是系统自带的应用。

包括 Desktop、Files、Market、Settings、Dashboard、Control Hub、Vault 和 Wise。

它们把文件、密码、知识、监控和系统管理做成用户可以直接使用的产品。

第五层是 Community apps。

这是社区应用层。

Olares Market 目前提供 200 多个应用。

它们覆盖 AI、生产力、媒体和智能家居。

具体数量会随 Market 变化。

开源不是单独的第六层。

它是贯穿整个系统的属性。

Olares 核心仓库采用 AGPL-3.0。

第三方组件遵循各自的许可证。

再看左边。

Olares OS 是右侧这套完整系统。

Clients 是用户的访问入口。

包括桌面端、移动端和浏览器扩展。

LarePass 是主要客户端。

Olares Space 管理域名、远程访问和备份等服务。

Olares Market 负责应用分发。

Olares CLI 和 Skills 让 Agent 用结构化接口管理系统。

自然语言理解主要发生在 Agent 一侧。

CLI 负责把操作变成稳定、可读取的系统命令。

【可删｜与 NAS 的关系】

传统 NAS 最擅长存储、磁盘和网络共享。

今天很多 NAS 也支持容器、虚拟机和应用中心。

Olares 并不否定这些能力。

它也不声称自己替代所有 NAS 功能。

它做的是另一层工作。

它把应用、身份、网络、Agent、中间件和算力统一成一个操作系统。

它也可以继续使用已有的 SMB、NFS 和 NAS 存储。

## 第 7 页｜应用沙盒

Olares 是一个多用户系统。

多个用户会共享同一台设备。

每个用户会安装自己的应用。

Agent 也会代表用户执行任务。

所以，系统需要隔离三个对象。

用户。

应用。

以及 Agent。

默认情况下，它们彼此隔离。

需要的能力，再通过 Manifest 明确授予。

这页可以用四个问题来理解。

谁可以行动。

应用怎样运行。

应用怎样连接。

应用可以看到什么数据。

第一部分是身份和授权。

Authelia 提供统一登录入口。

LLDAP 管理用户、群组和角色。

用户不需要在每个应用里重新建立一套身份。

工作负载权限使用 Kubernetes Role 和 ServiceAccount 管理。

文件、搜索和通信中的资源关系，使用 ReBAC 管理。

Agent 不会因为是 Agent，就自动获得整台设备的权限。

它只能在用户授权和应用能力允许的范围内行动。

敏感凭证也不直接放进应用。

目前使用 Infisical 管理 LLM API key 和 Agent cookie 等凭证。

OpenBao 是后续统一 secret management 的方向。

系统也预留了硬件密钥托管接口。

这部分的集成程度，要以实际版本为准。

第二部分是运行时和资源隔离。

每个应用有自己的运行环境。

不同用户的工作负载放在各自的 namespace 里。

应用默认使用非特权容器。

OPA 会在安装或运行前校验声明的权限。

CPU、内存和加速器需求，需要提前声明。

资源紧张时，系统会保护关键服务。

具体的驱逐和优先级策略，会随版本继续完善。

第三部分是网络边界。

Olares 支持四条访问路径。

第一条是互联网访问。

标准 Web 入口可以获得 Olares 域名和 HTTPS 证书。

也支持自定义域名和其他反向代理。

第二条是 LarePass VPN。

客户端使用 Tailscale。

控制面运行在用户自己的 Olares 上。

系统优先使用局域网或点对点连接。

必要时再使用 relay。

第三条是局域网访问。

`.local` 地址提供本地 HTTP 访问。

Overlay Gateway 支持设备发现、投屏等局域网协议。

第四条是集群内部通信。

这里使用服务身份、mTLS 和访问策略保护东西向流量。

部分服务网格能力仍在不同分支和集成路径中完善。

正式交付范围以对应版本为准。

第四部分是数据和文件边界。

Olares Files 把存储分成不同空间。

Home、Data、Cache 和 Common 有不同的所有权和持久性。

应用只能挂载经过授权的路径。

Home、Data 和 Common 可以进入跨节点持久数据面。

Cache 保留在本地。

它可以被重建。

Olares 也可以接入 Sync、云盘、USB、SMB、NFS 和现有 NAS。

Managed RAID 目前是下一阶段能力。

不能把它说成已经交付。

Files 还提供浏览、预览、编辑、移动、压缩和分享。

Search 负责发现候选内容。

Files 在真正执行操作前再次检查权限。

Agent 使用统一的 URL 访问不同存储后端。

它不需要为每一种存储单独适配 API。

贯穿这四部分的是同一条规则。

Manifest 负责声明。

OPA 负责校验。

平台负责执行。

这页描述的是完整产品方向。

其中一部分是当前能力。

一部分是未来几个月的目标架构。

下一页继续看硬件算力怎样被统一管理。

## 第 8 页｜异构 AI 算力管理

上一页讲隔离。

这一页讲算力。

本地 AI 硬件正在变得越来越多样。

有 Intel、NVIDIA 和 AMD 的独立显卡。

也有 Intel Core Ultra、NVIDIA GB10、AMD Ryzen AI Max 和 Apple M 系列这样的 SoC。

每种硬件有不同的驱动。

也有不同的运行时和资源模型。

如果每个应用都直接适配硬件，维护成本会很高。

Olares 在中间提供两层抽象。

第一层面向硬件。

我们叫它 Accelerator resources。

系统为不同设备启用 OpenVINO、CUDA 或 ROCm 等后端。

再通过设备插件，把加速器暴露成可调度资源。

系统统一设备标签、调度和注入方式。

也持续观测利用率和设备健康。

第二层面向应用。

我们叫它 Application allocation。

应用根据硬件和加速模式使用对应镜像。

用户在安装或启动时选择目标设备。

它可以是一张卡。

也可以是多张卡。

还可以来自同架构的多个节点。

系统通过 UI 显示分配结果和冲突。

当前版本已经支持安装和启动前的选择与绑定。

基于优先级的自动抢占、重平衡和扩缩容，仍然是产品方向。

不能把它理解成所有场景都已经完全自动化。

右边是模型和 AI runtime。

这里列出了语言、图像、视频、音频、OCR、3D 和 Embedding 模型。

也列出了 llama.cpp、vLLM、Ollama、ComfyUI 等运行环境。

这表示系统希望覆盖的模型类型。

不表示每个 runtime 在所有硬件上都有相同成熟度。

下半部分讲两个价值。

第一个价值，是统一生命周期。

系统负责驱动安装、设备发现、运行时配置、调度、绑定、监控和故障处理。

接入新设备时，设备相关改动尽量留在自己的模块里。

第二个价值，是资源可以组合。

可以使用单卡。

可以使用多卡。

可以使用跨节点资源。

也可以使用外接 eGPU。

开发者声明资源需求。

用户决定分配哪一块资源。

应用不用关心设备具体位于哪台机器。

这里也有一个边界。

目前跨节点成熟能力，主要是同架构集群和 NVIDIA 路径。

跨 AMD64 和 ARM64 的全自动资源池，仍然是下一阶段能力。

右下角是三种分配模式。

第一种是时间分片。

多个应用轮流使用完整 GPU。

它适合顺序工作流。

比如先用 LLM 写歌词。

再用音乐模型生成歌曲。

最后用图像模型生成封面。

如果显存不能同时放下三个模型，系统可以按步骤切换。

目前时间分片主要支持 NVIDIA 独立显卡。

第二种是显存分片。

系统把显存划成不同配额。

多个应用可以同时运行。

它适合 ComfyUI、语音服务和常驻模型服务。

第三种是独占。

一个应用占用整块加速器。

它适合渲染和游戏等负载。

切换模式时，已绑定的应用需要停止。

之后需要重新分配和恢复。

算力准备好以后，下一步是 Agent 怎样安全地使用这些能力。

## 第 9 页｜从人的意图到受治理的 Agent 行动

这页讲一条完整的 Agent 请求链路。

我们想解决四组关系。

人与 Agent 的关系。

Agent 与 Agent 的关系。

Agent 与软件的关系。

以及 Agent 与用户数据的关系。

第一层是输入设备。

手机、电脑、眼镜、音箱、耳机和手表，都可以成为入口。

它们把语音、文字、图像和事件送进系统。

第二层是 Personal Agent。

它可以是 Claude Code、OpenClaw、Hermes Agent、Codex 或 Olares Ashia。

Agent 可以运行在 Olares 上。

也可以运行在用户自己的其他设备上。

兼容的 Agent 通过 `olares-cli` 或 Olares Skills 调用系统操作。

这些操作是结构化的。

它们仍然受到身份和权限约束。

自然语言理解由 Agent 完成。

CLI 不绕过系统权限。

它也不等于远程主机管理的所有能力。

第三层是 Agent Orchestration。

请求真正执行前，会经过控制层。

AI Router 给应用提供统一的 AI 能力入口。

它可以在本地模型和远端模型之间路由。

也负责授权和用量策略。

Agent Sandbox 给 Agent 提供受控环境。

Agent 可以在里面使用工具、模型和上下文。

但它不能超出明确的边界。

Agent Collaboration 用于多 Agent 协作。

ACP 是我们采用的协议方向之一。

这不表示所有第三方 Agent 已经支持同一种协作协议。

Observability 负责记录用量、配额、权限、可靠性和执行状态。

Learning loop 使用用户反馈改进上下文和 Agent 行为。

这一项是方向性能力。

目前还不能说已经完整交付。

第四层是 Agent Capability。

它由 Tools、Models 和 Context 三部分组成。

Tools 让 Agent 能够行动。

数字环境包括 Chromium、redroid、Windows VM 和 macOS VM。

物理环境包括 Home Assistant 和机器人。

自动化工具包括 n8n 和 NocoDB。

Models 让 Agent 能够推理。

这里包括语言、翻译、图像、视频、OCR、3D、语音和检索模型。

Context 让结果和用户真正相关。

它包括文档、笔记、知识库和接入的 SaaS。

也包括邮件、聊天、联系人、日历、照片和健康数据。

Tools 负责行动。

Models 负责推理。

Context 提供与用户有关的事实。

第五层是 Olares Foundation。

它是每次 Agent 行动的共同底座。

包括沙盒、应用生命周期、中间件、算力、网络、身份、存储和可观测。

左边是请求怎样通过系统。

右边是三个设计原则。

第一个原则，是优先提供原生系统接口。

GUI Agent 通过像素、DOM 或 accessibility tree 模拟人的操作。

这种方式可以覆盖没有接口的软件。

但对长期、有状态的任务，它通常更慢，也更容易受界面变化影响。

Olares Skills 提供意图级工作流。

`olares-cli` 提供结构化、机器可读的系统操作。

Agent 可以读取 JSON、日志、事件和指标。

这样更容易判断操作是否成功。

我们的目标，是让 Agent 不只使用 Olares。

它也能参与开发、部署和调试 Olares 应用。

这还是目标。

不是已经全部完成的结论。

第二个原则，是每类 AI 能力提供一个相对统一的接口。

LLM 的 API 正在逐渐趋同。

但图像、音频、翻译、OCR 和视频仍然各有接口。

Olares 按领域统一 API、模型下载和推理引擎部署。

Agent 可以在同一类能力中切换不同模型。

这里不是说所有 AI 领域共用一个 API。

第三个原则，是先建立一个始终在线的上下文底座。

行业还没有形成公认的长期 Memory 方案。

所以 Olares 先做 Memory 的下一层。

先把分散在设备和服务里的数据，连接到一个长期在线的系统。

底层数据由用户保存。

用户以后可以尝试不同的 Memory 方案。

不需要把自己的全部上下文锁在单一 Agent 里。

很多 Agent 平台停在理解和回答。

Olares 希望把链路继续闭合。

让 Agent 在用户自己的基础设施上行动。

同时让这个行动可以授权、隔离和观察。

系统能力讲完以后，下一步是选择具体场景。

## 第 10 页｜七个已经被云端验证的场景

这页解释我们怎样选择场景。

我们使用两道筛选条件。

第一，用户已经在云端为这类需求付费。

第二，本地开源模型已经能在 24GB 级个人云上提供可用体验。

两项都满足，我们才优先做。

这张表有七个场景。

我逐行简单介绍。

第一行是 Personal Agent。

代表产品包括 OpenClaw、Hermes、Claude Code 和 Codex。

Claude Code 和 Codex 的增长，说明用户愿意为能够完成任务的 Agent 付费。

幻灯片引用了 Claude Code ARR 和 Codex 周活数据。

这些数字来自公司披露或二手报道。

正式使用前需要复核。

本地方案是 27B 级模型，加上 OpenClaw 或 Hermes。

24GB 显存可以运行合适的量化版本。

实际速度取决于上下文长度、量化方式和推理引擎。

Olares 的价值在于上下文。

一个能操作整套系统的 Agent，需要看到很多个人数据。

上下文越完整，越需要明确的数据边界。

Lovable ARR 约五亿美金，估值六十六亿。

Manus 的收购价约二十亿美金。

本地处理可以减少音频进入第三方服务器。

Midjourney 等产品已经证明用户愿意为生成图片付费。

这个速度取决于模型和工作流。

素材和人脸模型也可以保留在本地。

第五行是智能家居。

代表产品包括 Alexa、Google Home、Ring 和 Nest。

聊天记录也可以保留在自己的设备上。

实时数字人目前还是方向。

本地可以运行即时通信、文档协作和知识库问答。

这类场景里，企业数据和 AI 模型可以留在内网。

这有助于数据主权和成本管理。

企业仍然需要权限、审计、保留策略和组织流程。

## 第 12 页｜Arc Pro B70 外接 GPU

前面七个场景里，有四个很适合 NAS 类设备。

Personal Agent。

音频。

智能家居。

私人办公。

NAS 本来就长期在线。

它也已经保存了文件、录音和团队知识。

但很多 NAS 没有足够显存运行较大的本地模型。

我们的方案，是外接一块 Arc Pro B70。

B70 有 32GB 独立显存。

它可以通过 OCuLink、Thunderbolt 或高速 PCIe 连接。

Olares 负责设备接入、模型运行和 Agent 应用。

对已经有高速外接口的 NAS，厂商不需要先重做整机。

可以用外置模块验证 AI NAS 产品。

左边是一个硬件对比。

为了尽量公平，主表使用同一个模型。

模型是 Qwen3.6-35B-A3B。

测试使用单流、4-bit，并关闭 MTP。

给现有 NAS 增加 B70、扩展坞、线缆和供电，成本大约是 1100 到 1300 美元。

如果从零配置一整套，成本大约是 1600 到 2000 美元。

表里还比较了 AMD 395+ 迷你机、Mac Studio、DGX Spark 和桌面 RTX 5090。

这些价格是 2026 年 7 月的市场快照。

它们会变化。

性能方面，不需要把 B70 说成最快。

B70 的 decode 大约是每秒 55 到 82 个 token。

AMD、Mac 和 DGX Spark 的 no-MTP 结果，大多也在每秒 50 到 100 个 token 的范围。

桌面 RTX 5090 更快。

但表里的价格只是一张显卡。

不包括主机和电源。

表格最后一行是 DGX Spark 的优化结果。

它使用 NVFP4、vLLM 和 MTP。

测试条件和主表不同。

所以不能直接横向比较。

对 Personal Agent 来说，prefill 也很重要。

Agent 每轮都要处理系统指令、对话、记忆、文件和工具结果。

Prefill 决定用户多久看到第一个字。

Decode 决定开始回答以后说得多快。

B70 当前 prefill 大约是每秒 615 个 token。

这个结果还有提升空间。

我们认为主要瓶颈在 XMX attention 和推理软件栈。

但在完成更多验证前，也不能简单断言硬件已经达到什么上限。

右边有六个产品价值。

第一，外接对大模型稳态推理的影响较小。

模型权重加载进显存以后，主要计算发生在 GPU 内部。

现有测试里，外接比内置 PCIe x16 低约百分之二到三。

这个结论适用于对应的大模型推理测试。

不能外推到游戏或所有 GPU 任务。

第二，升级更灵活。

用户以后可以只更换 GPU 模块。

不一定要更换主机。

外置 GPU 也可以在兼容设备之间迁移。

第三，OEM 的产品工程更简单。

B70 参考功耗约 230 瓦。

很多消费 NAS 的整机电源只有 60 到 250 瓦。

如果把 GPU 放进机箱，可能需要重做电源、散热和结构。

外置方案把这部分工程放在扩展坞里。

第四，一套硬件可以覆盖四类负载。

也就是 Personal Agent、音频、智能家居和私人办公。

第五，对 Intel 来说，每台 AI NAS 都可能成为 Arc Pro 的新设备入口。

这是一种合作机会。

不是已经发生的销量。

第六，当前进入成本较低。

给已有 NAS 加算力，大约需要 1100 到 1300 美元。

这低于表里的多数完整整机。

这条路线也有两个问题。

第一个问题，是 prefill 软件性能仍需优化。

第二个问题，是当前价格优势受到统一内存价格周期影响。

如果内存价格回落，差距会缩小。

所以，长期价值不能只建立在今天的价格上。

更稳定的价值是复用已有 NAS。

数据不需要迁移。

GPU 可以模块化升级。

这页的结论不是 B70 每项跑分第一。

结论是，它提供了一条进入成本较低的 AI NAS 路线。

Olares 可以把硬件、模型和应用整合成产品。
