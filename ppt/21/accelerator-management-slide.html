One lifecycle, composable across every accelerator.

01 · Manage accelerator lifecycle

One interface. Olares abstracts the whole lifecycle—from host driver installation and device discovery to runtime setup, scheduling, binding, monitoring and fault eviction.

Module-local integration. A new device implements the lifecycle interface inside its own module, instead of scattering device-specific changes across the system.

02 · Compose resources

Any topology. Compose one resource, multiple instances, mixed resource types, cross-node capacity or an external eGPU.

Two-layer binding. Developers declare resources in software; users choose and allocate them through the system.

Placement-independent apps. The app contract stays stable from one device to multi-GPU or same-architecture pools.

Three allocation modes—transparent to apps, with no code changes.

Time slicing

Why the name: Apps take turns using the full GPU. Best for sequential workflows where each model can consume most of its memory.

Album example: LLM lyrics → ACE-Step 1.5 XL music → Z-Image cover. Without Olares, users stop, release and re-bind the GPU between stages—and repeat this for every track. Time slicing rotates the GPU automatically, so the album feels like one continuous workflow.

NVIDIA dGPU only

Memory slicing

Why the name: VRAM is split into fixed quotas, so apps run concurrently. Best for: controlled resident apps such as ComfyUI, Speaches or model servers.
