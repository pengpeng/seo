Sandboxes Every User, Application and Agent

One multi-user device — users, applications and Agents run side by side, isolated by default and granted only what they declare.
01 · Identity & authorization

Who can act—and on whose behalf

Unified authentication, workload authorization and secrets custody across users, applications and Agents.

Unified identity

One sign-in across the system. Authelia gives every user and application a consistent authentication entry.

Roles stay consistent everywhere. LLDAP maintains users, groups and Admin / Member roles across applications.

ReBAC & Agent delegation

Every workload receives only what it needs. Kubernetes Roles and ServiceAccounts bind permissions to each application.

People and groups keep their own boundaries. ReBAC scopes Search, Files, IM and Agents by user, group and resource.

Secrets & key custody

Sensitive credentials stay out of applications. Infisical manages LLM API keys and Agent cookies today; OpenBao comes next.

Key custody can move into hardware. A low-level HSM interface supports third-party HSM integration.

03 · Network boundary & access

How each application connects

Four access paths, protected by declared entrances and layered security.

Public networkInternet · reverse proxy

Domains and HTTPS come ready. Every Web entrance gets a free Olares domain and managed certificate, with custom domains and user-selected reverse proxies also supported.

Private networkVPN · LarePass

Private access without a third-party control plane. Tailscale clients use Headscale hosted on Olares, preferring LAN or direct P2P and falling back to relay.

Local networkLAN · .local HTTP + Overlay

Local compatibility stays intact. .local HTTP serves Web access; per-app Overlay carries discovery, casting and native LAN protocols.

Platform networkIn-cluster · Linkerd

Identity-aware service mesh. Automatic mTLS and ServiceAccount identity secure every workload; fine-grained policy, metrics and audit signals make east-west traffic observable and governable.

02 · Runtime & resource isolation

How each application runs

Isolated runtimes, reduced privileges and explicit resource envelopes.

Runtime boundary

Each application gets its own runtime. Applications do not share an execution boundary with the host.

Users remain separated. User-scoped namespaces keep workloads and service identities inside their owner's space.

Reduced privilege

Privilege starts low. Applications run in unprivileged containers instead of receiving host-level access.

Policy is checked before execution. OPA admission validates declared permissions before installation or execution.

Resource envelope

Capacity is declared up front. CPU, memory and accelerator requests tell the scheduler what an application needs.

The system protects itself under pressure. Olares intelligently evicts workloads to preserve stability and keep critical services running.

04 · Data & filesystem boundary

What data each application can see

Olares Files applies declared scopes across persistent, connected and Agent-ready storage.

Sandboxed spaces
Every path has an owner. Home, Data, Cache and Common define ownership, durability and sharing; applications mount only declared paths.
Cross-node data plane
Persistent data follows the cluster. Home, Data and Common remain available across nodes; Cache stays local and safely rebuildable.
Connected storage
Existing data stays where it is. Sync, cloud drives, USB, SMB, NFS and existing NAS join one file world. Managed RAID next.
PC-grade Files & sharing
Users keep familiar file operations. Browse, preview, edit, move and archive, then share through users, public links or SMB.
Search
Discovery comes before action. Search finds candidates by name, full content or image; Files verifies permission and acts.
Unified access URLs
Agents use one address model. Files exposes data through consistent URLs across local, distributed, cloud and external storage.
