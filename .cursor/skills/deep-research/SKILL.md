# 深度调研 Deep Research

给定一个主题（技术/市场/政策/公司/赛道），用**并行子代理 + 多源交叉验证**产出一份可溯源的研究报告：每个事实都能追到某个来源，来源分级、标注时效，并做强制反向复核。改编自 daymade V6（Claude Code），已适配 Cursor 工具。

模板与阈值速查见 [reference.md](reference.md)。

## 与其他 skill 的分工（不重叠）

- 品牌/SaaS 的 **Semrush 流量/关键词/竞品数字** → [brand-seo-research](/Users/pengpeng/seo/.cursor/skills/brand-seo-research/SKILL.md)
- **开源模型**的 SEO/部署词 → [model-seo-research](/Users/pengpeng/seo/.cursor/skills/model-seo-research/SKILL.md)
- **deep-research** 负责非"纯 Semrush 数字"的深度综合（理解一个领域/公司/技术、行业综述、技术选型、政策/竞品格局）。需要精确搜索量时，可在某个子任务里调用 Semrush MCP（`user-semrush`）或直接引用上述两个 skill 的既有报告。

## 工具映射（Cursor）

| 用途 | 工具 |
|------|------|
| 网络搜索 | `WebSearch` |
| 抓取全文 | `WebFetch`（DEEP 任务用；被 Auto-review 挡下时按提示重试） |
| 并行子代理 | `Task`：抓取调查用 `subagent_type: explore`，综合/复核用 `generalPurpose`；`run_in_background: true` 并行，**每组最多 3 个并发** |
| 精确搜索量（可选） | MCP `user-semrush` |
| 落盘 | `Write` / `Read`（笔记与报告都写文件） |

> Lead Agent（你）只读子代理写出的**蒸馏笔记**，不读原始搜索结果——原始网页内容留在子代理自己的上下文里丢弃，省 60–70% 上下文。

## 架构

```
Lead Agent（协调，尽量少吃原始搜索上下文）
  P0 预检 + 来源策略
  P1 研究任务板（角色 / 查询 / 并行分组）
  分派 ─→ 子代理 A ─→ 写 task-a.md ┐
       ─→ 子代理 B ─→ 写 task-b.md ┤（并行，≤3/组）
       ─→ 子代理 C ─→ 写 task-c.md ┘
                 research/<slug>.notes/ ←┘
  P2 汇总；P3 建引证登记（来源类型+时效+权威分+质量门）
  P4 证据映射大纲（标反向主张）；P5 依据笔记起草（不看原始搜索）
  P6 强制反向复核（≥3 个问题）；P7 溯源核验；成稿带置信度标记
```

## 模式选择（开工前先定）

| 维度 | 选项 |
|------|------|
| 主题模式 | **通用调研**（行业/政策/技术/赛道）｜ **公司专项**（单个企业，走六维+框架化分析，见 reference.md） |
| 深度模式 | **Standard**（5-6 任务，3000-8000 字）｜ **Lightweight**（3-4 任务，2000-4000 字） |

- 单一实体/概念、问题 <30 字 → Lightweight；多实体对比或用户说"深入/全面/comprehensive" → Standard。

## 来源治理

**可获取性分级**（每个来源都要判）：`public`（公开，✅）｜`semi-public`（需注册，✅注明）｜`exclusive-user-provided`（用户显式提供的付费订阅/私有 API，用于调研第三方，✅当独家优势）｜`private-user-owned`（用户调研自己时访问自己的私有账户，❌ 循环验证禁止）。

- **循环验证禁令**：不要用用户自己的私有数据去"发现"他本来就知道的关于自己的事；调研用户自己的公司/资产时，只从**外部研究者视角**用公开源（公开 WHOIS、新闻等）。
- **独家来源**：用户明确说"用我的 Crunchbase Pro 查竞品"时，接受为 `exclusive-user-provided` 并当优势用；无公开等价源则标 `[unverified]` 或略去该主张。

**来源类型标签**：`official`/`academic`/`secondary-industry`/`journalism`/`community`/`other`。

**质量门（摘要，完整阈值见 [reference.md](reference.md)）**：Standard ≥12 approved 源、≥5 独立域名、官方源占比 ≥30%、单一来源占比 ≤25%；Lightweight ≥6 源、≥3 域名、官方 ≥20%、单源 ≤30%。

## AS_OF 时效

P0 显式设定 `AS_OF = 今天(YYYY-MM-DD)`。每条时效性主张都带来源发布日期；来源过旧（研究 >3 年、快变主题新闻 >6 个月）就降级置信度并在登记里标注。

## P0 · 预检与策略

先检查能力，再定策略变量：

| 检查 | 要求 | 缺失影响 |
|------|------|----------|
| `WebSearch` 可用 | 必须 | 不能继续 |
| `WebFetch` 可用 | DEEP 任务必须 | 退化为 SCAN-only |
| `Task` 子代理 | 优先 | 退化为顺序执行（Lead 自己扮演各角色） |
| 可写文件 | 必须 | 退化为内存笔记 |

设定：`AS_OF`、`MODE`（Standard/Lightweight）、主题模式（通用/公司专项）、`反向复核计划`（打算检验哪个相反解读）。
报告：`[P0 完成] 子代理:{是/否} 模式:{...} AS_OF:{...}`。

## P1 · 研究任务板

把研究问题拆成 4-6 个（Standard）/ 3-4 个（Lightweight）调查任务。每个任务：

- **专家角色**（如"生态图谱分析师""政策史学者"）
- **目标**：一句话
- **查询**：2-3 条预设 query
- **深度**：DEEP（抓 2-3 篇全文）或 SCAN（摘要够用）
- **输出路径**：`research/<slug>.notes/task-{id}.md`
- **并行分组**：A 组（相互独立）或 B 组（依赖 A 组结论）

规则：每个任务是一个专家会独立负责的连贯子题；A 组任务彼此独立且来源多样；每个并行组 ≤3 任务；标出时效敏感与引用老化风险。
公司专项模式下，任务板对齐六维（公司基本面 / 业务与产品 / 竞争位置 / 财务与运营 / 近期动态 / 内部或专有源），见 [reference.md](reference.md)。
报告：`[P1 完成] {N} 任务 {M} 组，分派 A 组`。

## P2 · 分派与调查

用 [reference.md](reference.md) 的**子代理任务提示模板**填好变量，`Task` 分派子代理（`run_in_background` 并行，A 组 ≤3 并发）。每个子代理搜索、抓取、给每个来源打 `Source-Type` + `As Of` + 权威分，然后**只把蒸馏后的笔记**写进 `task-{id}.md`（格式见 reference.md）：Sources / Findings（≤10 条，每条一句带来源号）/ Deep Read Notes（DEEP 任务 2-3 篇全文精读）/ Gaps（搜了但没找到的、以及至少一个相反主张候选）。

A 组全部完成后再派 B 组（B 组可读 A 组笔记）。无子代理时 Lead 顺序扮演各角色，原始搜索结果写完笔记即弃。
报告：每任务 `[P2 task-{id} 完成] {N} 源 {M} 发现`；全部 `[P2 完成] 共 {N} 任务 {M} 源，开始建登记`。

## P3 · 引证登记 + 来源质量门

Lead 读**所有** task 笔记，建统一登记（格式见 [reference.md](reference.md)）：合并去重（按 URL，同 URL 取权威分高者）→ 按首次出现分配**最终 [n] 编号** → 标 source_type / as_of / 权威分(1-10) / 来源任务 → **套质量门**，低于阈值的源明确列进 Dropped 并说明原因。

- **这些 [n] 是最终编号**：P5 只能引 Approved 列表里的源；Dropped 的源永不再出现。
- 信息黑箱（查无公开足迹的实体）：如实写"外部研究者无法验证"，记录所有失败的检索尝试，**不要**用用户私有凭证补全、**不要**凭域名注册就断定存在、**不要**用推测填空。

报告：`[P3 完成] {approved}/{total} 源，{N} 域名，官方占比 {xx}%`。

## P4 · 证据映射大纲

Lead 读笔记+登记，**按主题（而非任务顺序）**设计章节，每节映射到具体 findings 的来源号，标出需反向复核的节、时效敏感主张。大纲格式见 [reference.md](reference.md)。

## P5 · 依据笔记起草

按 [reference.md](reference.md) 的报告模板逐节写。规则：每个事实性论点带 [n]；数字/百分比必须有源；每节加**置信度标记**（High/Medium/Low + 依据）；证据冲突时给一句**反方解释**；**不得引入新源**；无支撑的陈述标 `[unverified]`。
反幻觉：Lead **绝不编 URL**（只用子代理笔记里的）、**绝不编数字**（笔记没有就标 `[unverified]`，精度不超过来源）。
报告：`[P5 进行中] {N}/{M} 节，约 {字数} 字`。

## P6 · 强制反向复核

对每个主要结论做相反视角检查：结论会不会是错的？哪些高影响主张只靠单一来源？哪些主张缺官方/学术支撑？时效主张是否用了过旧的源？**至少找出 3 个问题**（找到 0 个就再查一遍）。可用 `Task`（`generalPurpose`）派并行复核子代理（主张校验 / 来源多样性 / 时效 / 矛盾发现）再汇总。
产出并入报告的"核心争议"小节。报告：`[P6 完成] {N} 问题：{critical}严重 {high}高 {medium}中`。

## P7 · 溯源核验

定稿前交叉核对：报告里每个 [n] 对照 Approved 登记；抽查 ≥5 条主张能追到 task 笔记；移除/修正无法溯源的主张；确认没有 Dropped 的源"复活"；检查关键主张的来源集中度。
报告：`[P7 完成] {N} 处抽查，修正 {M} 处违规`。

## 输出约定

- 最终报告：`research/<slug>.md`（`<slug>` 为主题短横线小写；无日期前缀，git 记录变更；同题直接覆盖更新）。抬头用 reference.md 报告模板的元信息行（日期/来源数/字数/模式/AS_OF/官方源占比）。
- 单次运行的子代理笔记与登记：`research/<slug>.notes/`（`task-*.md`、`registry.md`），保留供溯源。
- 语言与术语：跟随用户语言，技术术语保留英文；参考文献一律带 Source-Type 与 As Of。
- **重写已有报告前**：先读现有同名报告，保留并去重旧洞见，不认可的旧结论明确指出——别把有价值的历史判断冲掉（与两个 SEO skill 同一规矩）。

## 反模式（别犯）

单遍起草不做多遍完整稿；按章节切遍数而非整篇；忽略用户给定的格式/模板；主张无引用；混用矛盾日期不指出；Lead 去读原始搜索结果（只读子代理笔记）；编 URL/编数字；让 Dropped 的源复活；时效主张漏 AS_OF；跳过 P6 反向复核；循环验证（用用户私有数据"发现"他自己已知的事）；用户给了独家付费源查竞品却不用。
